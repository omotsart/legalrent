# API-контракт: фасад и схемы данных

Публичный фрагмент, показывающий **контракт** библиотеки — сигнатуры и Pydantic-схемы
без реализации. Тела функций заменены на `...`. Данные в примерах вымышлены.

Контракт устроен вокруг двух решений:

- **Наружу видны ровно четыре имени.** Всё версионное (`route_rule_v10`, `_run_sql_v8`)
  скрыто за фасадом — вызывающий код не зависит от внутренних версий и их откатов.
- **Данные пересекают границу только как типизированные структуры.** `ask()` возвращает
  `Answer`, а не строку; маршрут — объект `RouteDecision` с полем `rule_name`, а не
  строка `"sql"`. Ответ можно проверить программно: какое правило сработало, какие
  фрагменты найдены и с какой релевантностью.

## Схемы данных (Pydantic)

```python
class RouteDecision(BaseModel):
    target: str               # "sql" | "rag" | "both" — куда направлен запрос
    rule_name: str            # какое правило приняло решение (для трассировки)
    matched_on: str = ""      # маркер/триггер, на котором сработало правило


class Chunk(BaseModel):
    source: str = ""                     # идентификатор документа-источника
    clause_index: Optional[int] = None   # номер пункта договора, если распознан
    filename: str = ""                   # имя файла-источника
    score: Optional[float] = None        # релевантность (после слияния RRF может быть None)


class Answer(BaseModel):
    text: str                            # текст ответа для пользователя
    route: RouteDecision                 # решение роутера — ОБЪЕКТ, не строка
    route_final: str                     # итоговый маршрут (виден fallback sql→rag)
    chunks: list[Chunk] = []             # найденные фрагменты с релевантностью
    sql_result: Optional[str] = None     # результат SQL-ветки, если была
    trace_id: str = ""                   # связь со строкой трассировки
    latency_ms: int = 0                  # латентность вызова
    model: str = ""                      # какой провайдер сгенерировал ответ
    entities: dict = {}                  # извлечённые сущности (напр. арендатор)
    guard_fired: bool = False            # сработал ли детектор галлюцинаций


class ClauseParams(BaseModel):
    category: str                        # категория playbook (или "out_of_model")
    params: dict[str, Any] = {}          # параметр→значение, извлечённое из правки
    source: Literal["rule", "llm", "mixed", "none"] = "rule"  # чем извлечено
    raw_text: str = ""                   # фрагмент правки, из которого извлекали
```

## Почему контракт такой

- **`route` — объект, а не строка.** Ответ несёт не только «куда пошёл запрос», но и
  *почему* (`rule_name`, `matched_on`). Маршрут становится аудируемым и тестируемым по
  структуре, а не по тексту ответа.
- **`Chunk.score` опционален.** После слияния лексической и векторной веток по RRF единый
  score определён не всегда — поле честно допускает `None`, а не выдумывает число.
- **`ClauseParams.source` фиксирует происхождение значения** (правило / LLM / смешанно).
  В оценке правок цвет вердикта выносит код по `baseline` из playbook, а не LLM — и по
  этому полю видно, откуда пришло каждое значение.
- **Возврат `Answer`, а не текста.** Тот же вызов обслуживает и UI, и eval, и отладку:
  структура позволяет проверять контракт (маршрут, фрагменты, арифметику) без разбора строк.
